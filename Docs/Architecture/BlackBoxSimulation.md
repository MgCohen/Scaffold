# Black Box Simulation Architecture

This document outlines the **Black Box Simulation** architecture for game development, based on the concepts presented in the video "Brian Cronin: Black Box Sim for Roguelikes".

## Core Concept
The foundational principle of this architecture is the **strict decoupling of game logic (the Simulation) from the game engine (the Presentation/Rendering/Input layer)**. The simulation is a "black box" that takes inputs and produces state changes without any awareness of how those changes are visually represented to the player.

## Key Components

### 1. World State
The `WorldState` is a purely data-driven container representing the entire state of the game at any given moment.
* **Data Types**: It strictly relies on integers (avoiding floating-point math to guarantee cross-platform determinism) and is keyed by entities.
* **Contents**: It encapsulates everything needed to reconstruct the game, including the RNG state, unit statistics (HP, defense), and equipment.

### 2. World Command
Player actions and simulation inputs are encapsulated into `WorldCommand` objects (e.g., `HealUnitCommand` or `MoveUnitCommand`).
* **Self-Contained & Serializable**: Each command is fully serializable, making them the payload for networking and save files.
* **Resolve Function**: They possess a fundamental `resolve(WorldState)` function that mutates the provided state.
* **Rule of Isolation**: Commands are completely isolated; they never invoke or call other Commands. Any shared logic must reside in static helper functions.

### 3. The Black Box (Simulation)
The simulation logic engine that processes a stream of `WorldCommand`s against the current `WorldState`. Because it strictly controls execution, it is entirely deterministic: applying the same sequence of Commands to the same initial State will consistently produce the exact same resulting State.

### 4. The Presentation Bridge
To connect the headless simulation to the game engine:
* **Events & Notifications**: When the Black Box processes a command and mutates the state, it emits events (e.g., `"Unit Healed"`, `"Damage Taken"`).
* **Unidirectional Flow**: The Simulation *never* queries or reads from the Presentation side. The Presentation layer (VFX, animations, UI, audio) merely listens to these state-change events and updates the visual representation accordingly.

## Architectural Benefits

Adopting the Black Box Simulation provides multiple long-term advantages, particularly for complex games or commercial roguelikes:

1. **Determinism**: Essential for consistent behavior, ensuring that edge cases and bugs are reproducible.
2. **Simplified Save/Load System**: Because the entire `WorldState` is purely data, saving the game simply involves serializing the current state object.
3. **Optimized Networking**: For multiplayer games, you do not need to sync massive state objects; you only need to transmit the small `WorldCommand` objects. Each client's local Black Box will independently arrive at the same state.
4. **Replay System**: Replays can be generated trivially by storing the initial RNG seed and the sequence of Commands executed during the run.
5. **Undo/Redo Capability**: The architecture naturally supports taking structural snapshots of the `WorldState`, making an "Undo" feature simple to implement.
6. **Automated & Headless Testing**: You can run thousands of automated tests on the game logic without incurring the overhead of loading graphics, audio, or the full engine environment.

## Context & Applicability
While this pattern introduces initial overhead that might be excessive for short game jams, it is highly recommended for long-term, scaleable projects (like *Inkbound* or *Monster Train*) where maintainability, networking, and robust logic testing are critical to success.
