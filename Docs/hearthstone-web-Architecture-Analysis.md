# hearthstone-web Architecture Analysis

This document outlines the architecture, messaging, and input systems of the `hearthstone-web` repository.

## 1. Core Architecture & Game State

*   **Design Paradigm**: `hearthstone-web` is a purely front-end, vanilla JavaScript browser application. It does not use a modern component framework (like React or Vue) or an ECS (Entity Component System) game architecture.
*   **State Management**: There is no authoritative, decoupled Game State object or State Machine. Game state is stored globally in JavaScript variables and directly intertwined with the HTML Document Object Model (DOM). For example, the state of a minion's health is often read directly from or written directly to the `innerHTML` of the corresponding DOM element.
*   **Procedural Logic**: Rules and effects are handled procedurally through standard JavaScript functions (found in files like `attack.js` and `card_effects.js`). 

## 2. Client-Server Communication

*   **Network Layer**: **None**. `hearthstone-web` has no backend or server component. It operates entirely locally within the user's browser for a single-player experience against a scripted AI (handled in `AI.js`). 
*   Because there is no network layer, there are no sockets, APIs, standard `NetworkMessage` packets, or server architectures to analyze.

## 3. Messaging and Block Logic

*   **Messaging System**: Because the game runs procedurally in a single thread with no networking or rules engine, there is no generic message bus or event queue like SabberStone's `TaskProcessor` or `PowerHistory`.
*   **Effects Resolution**: Actions execute synchronously from top to bottom. If a card is played, the `card_effects.js` function for that specific card runs immediately and mutates the DOM elements directly. There is no concept of complex nested effect blocks or event resolution phases.

## 4. Player Input & Client Prediction

*   **Input Translation**: Player input is caught via standard DOM Event Listeners (`onclick`, `mousedown`, `mousemove` in `elementsController.js` and `pack_handler.js`). The raw click event resolves immediately into a hardcoded game function (e.g., clicking the "End Turn" button triggers an `endTurn()` function). There is zero intermediate "Task" abstraction separating UI interactions from game logic modifications.
*   **Client Prediction**: Because the application is a monolithic offline frontend, client prediction is entirely unnecessary and inapplicable. The "client" simply executes the rules authoritative locally.

## Summary
`hearthstone-web` is a simplistic, monolithic JavaScript web frontend replicating the visual behavior of Hearthstone. It lacks any formalized rules engine structure, messaging queues, or client-server architecture. All state is tightly coupled to the DOM, and all actions are resolved immediately and synchronously via procedural methods.
