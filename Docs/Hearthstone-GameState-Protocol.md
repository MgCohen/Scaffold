# Hearthstone Game State Protocol & Architecture

Based on the documentation from HearthSim about the Hearthstone Game State Protocol and HSReplay, this document details how Hearthstone manages data and game state between the server and the client.

## 1. The Hearthstone Game State Protocol

At its core, a Hearthstone game is represented as a **bucket of entities**. Each entity is a key-value store of integer properties known as **Tags** (specifically `GameTag`).

### Core Concepts

- **Tags (`GameTag`):** All entity properties are key-value integer pairs. A tag not existing is fundamentally identical to its value being 0. There are over 500 tags, but the server only sends the client the tags it needs to know about.
- **Entities:** Every key-value store instance in the game is an entity. There are three types:
  1. **Game:** Always has Entity ID `1`.
  2. **Players:** Always two of them, taking Entity IDs `2` and `3`.
  3. **Cards:** Literally everything else in the game (Minions, Spells, Weapons, Enchantments/Buffs, Heroes, and Hero Powers).
- **Zones:** From its creation, an entity always exists in a zone. There are 7 zones: `PLAY` (the physical battlefield/active area), `DECK`, `HAND`, `GRAVEYARD`, `SECRET`, `SETASIDE` (used for mechanics like Discover), and `REMOVEDFROMGAME`. 
  - *Note:* Game mechanics are just transitions between these zones. For example, discarding is a transition from `HAND` to `GRAVEYARD`, drawing from `DECK` to `HAND`, and bouncing from `PLAY` to `HAND`.

### The Game's Lifecycle & Network Packets (Powers)

Since the game is just state, events are executed by mutating the global state via **State Packets** (sometimes called "Powers").

- **Packet Types (`PowerType`):** 
  - `CREATE_GAME`: Sets up the base game and players.
  - `FULL_ENTITY`: Creates a new entity with initial tags.
  - `TAG_CHANGE`: Mutates a single tag on an entity.
  - `SHOW_ENTITY` / `HIDE_ENTITY`: Used to reveal/hide data from a player to prevent cheating.
  - `BLOCK_START` / `BLOCK_END`: Used to chunk sequences of events together.
  - `META_DATA`: Non-mutative, primarily used for correctly rendering animations, damage, and targets on the client side.
- **The Input Loop:** The server entirely determines what is playable by sending "Options" to the current player. When a player takes an action (like playing a card), they send a `SendOption` packet to the server, which simulates the outcome, applies State Packets, and evaluates new options.
- **Action Blocks:** Complex interactions (e.g., playing a card that draws another card, which triggers a knife juggler, which kills a minion, which triggers a deathrattle) are wrapped in `BLOCK_START` and `BLOCK_END` tags. This creates a nested hierarchy of actions so client animations follow the exact chain of causality.
- **Imperfect Knowledge:** To prevent cheating, the client and the server have asymmetrical knowledge. While the server tracks global state with `FULL_ENTITY`, it strips out hidden tags before sending them to the client. When a card is drawn and revealed, the server sends a `SHOW_ENTITY` packet to update the client's local known state.

## 2. HSReplay (Hearthstone Replays)

HSReplay is an open, standardized XML format used by the community to save and share Hearthstone replays, built directly on top of the Game State Protocol.

- **Structure:** An XML replay serves as a sequential, chronological log of all Network/Power packets sent by the server. 
- **Representation:** Elements strictly follow the protocol outlined above:
  - It begins with `<Game>`, `<Player>`, and `<FullEntity>` setup blocks containing `<Tag>` children for initial state.
  - The timeline then unfolds sequentially tracking precise timestamps using tags exactly mapped to `PowerTypes` like `<TagChange>`, `<Block>`, `<ShowEntity>`, and `<Options>`.
- **Tooling:** Because this matches the internal packet structure of Hearthstone 1:1, tools and simulators (like python-hsreplay, Joust HTML visualizer, and HearthSim's ecosystem) can effortlessly rebuild any discrete moment of the game perfectly by walking through the XML packet by packet.

## Architectural Summary

Hearthstone's architecture is not built around telling the client imperative visual actions (e.g., "The minion attacks and it takes 3 damage"). Instead, it is designed around a strongly validated, server-authoritative **Finite State Machine**. 

The server simply sends incremental data differentials (Tag changes, Zone movements, Entities appearing/disappearing) wrapped in blocks, while strictly holding back secret state (like deck ordering). The front-end Unity client acts primarily as a dumb renderer that translates those pure data adjustments into visual models, sounds, and animations.
