# Fireplace Architecture Analysis

This document outlines the architecture, messaging, and input systems of the `fireplace` repository, a Hearthstone simulator written entirely in Python.

## 1. Core Architecture & Game State

*   **Python Call-Stack Engine**: Unlike SabberStone's or Brimstone's distinct `TaskQueue` systems, Fireplace leans heavily into Python's native call-stack to resolve game sequences. The `BaseGame` class manages the field (`entities`, `hands`, `board`).
*   **Triggers & Events**: The game resolves sequences using hardcoded sequences (`action_block`, `queue_actions`, `trigger_actions`). When an action occurs (like playing a card or dealing damage), the engine broadcasts an `Event` that checks for aura updates, secrets, or deathrattles recursively. Wait states are handled naturally as Python functions return.
*   **Object-Oriented Cards**: Every card in the game (e.g. `cards/witchwood/warlock.py`) is implemented as an explicit Python Class containing specific overrides (e.g., `def play(self):`).

## 2. Client-Server Communication

*   **TCP `Kettle` Server**: Fireplace includes a dedicated, basic network server within the `kettle/kettle.py` script.
*   **Connection**: It implements Python's built-in `socketserver.TCPServer`.
*   **Protocol**: Packets are sent over raw TCP. The server expects a 4-byte integer header indicating the payload length, followed by a raw UTF-8 string payload formatted strictly as **JSON**.

## 3. Messaging and Block Logic

*   **Deltas over Blocks**: Fireplace does not use a strict "PowerHistory BlockStart/BlockEnd" schema to broadcast data to a client. 
*   **JSON Serialization**: Instead, it maintains a localized `game_state` dictionary on the server. As the Python engine mutates the game entities, the `KettleManager` fires functions like `tag_change`, `new_entity`, and `show_entity`. These events package the diffs via a `KettleSerializer` into JSON blobs and stream them back to the connected client.

## 4. Player Input & Client Prediction

*   **Input Translation**: 
    1.  The Python server keeps a list of currently valid options for the priority player (the `options` array).
    2.  The client sends a JSON payload containing an `Index` (pointing to the option array) and a `Target` (the ID of the target entity).
    3.  `Kettle.read_packet` decodes the JSON.
    4.  `KettleManager.process_send_option` intercepts the JSON dictionary. It translates the index back to the server-authoritative valid `option`. 
    5.  It evaluates the option type (`END_TURN`, `DEFAULT`, `POWER`) and subsequently calls the underlying Python game engine directly (e.g., `self.game.end_turn()` or `entity.play(target)`).
    *   This successfully prevents hackers from sending arbitrary commands; they can only send integer index references to pre-approved Server options.
*   **Client Prediction**: Because the Fireplace repository only contains the Python `Kettle` server codebase and not an associated client frontend, client prediction is completely out-of-scope for the repository. The server expects the client to function as a passive graphical shell that visualizes the JSON gamestate streams.

Fireplace is an idiomatic Python implementation of a Hearthstone simulator. It eschews task queues for generic call stacks and resolves state locally.
