# Brimstone Architecture Analysis

This document outlines the architecture, messaging, and input systems of the `Brimstone` repository. Brimstone is designed primarily as an extremely fast, headless simulation and AI evaluation engine for Hearthstone, rather than a full game client/server setup.

## 1. Core Game State & Architecture

*   **Tree Search Focused**: Brimstone is uniquely designed around game-state branching for AI tree search (e.g., Depth First Search, Monte Carlo Tree Search). The `Game` state is heavily optimized to be cloned and branched thousands of times per second.
*   **Entity and Tags**: Like SabberStone, Brimstone models everything (Heroes, Minions, Spells, Weapons, Players) as an `Entity` containing Hearthstone `GameTags`.
*   **`ActionQueue` Engine**: Instead of a strict state-machine step loop, Brimstone evaluates the game entirely through its `ActionQueue`. 
    *   Actions (like dealing damage or drawing a card) are pushed onto the queue.
    *   The queue can evaluate synchronously (`ProcessAll()`) or asynchronously (`ProcessAllAsync()`).
    *   The `ActionQueue` allows injecting "Action Blocks" which effectively create the stacked resolution needed for complex interactions (like cascading Deathrattles).

## 2. Client-Server Communication

**Brimstone does not contain a network layer.** It is strictly a local simulation library. 
There are no TCP sockets, RPCs, or serialization mechanisms for sending data across a network. It is meant to be wrapped by a separate server application if multiplayer is desired.

## 3. Messaging and Block Logic (`PowerHistory`)

While Brimstone doesn't send messages over a network, it heavily tracks game state modifications using a system remarkably similar to the Hearthstone protocol in order to generate diffs for AI evaluation.

*   **`PowerHistory`**: Every action processed by the game logs an entry into `PowerHistory`.
*   **Block Grouping**: The ActionQueue naturally handles blocks. When a sequence of dependent actions begins, it starts an event log block, and closes it when the queue finishes that sequence.
*   **Delta Crunching**: Because Brimstone is an AI simulator, its `PowerHistory` implementation includes a very specific mathematical feature: `CrunchedDelta`. It compares two branch paths of a game and determines if the resulting sequence of `TagChange` events are *functionally equivalent*, allowing the AI engine to prune identical board states.

## 4. Player Input & Client Prediction

*   **Player Input**: Input is managed entirely via direct API calls into the `Game` representation. There is no concept of a "Game Client" sending a "Request" to a server. A user (or AI) script directly invokes commands on the entities in the local simulation.
*   **Client Prediction**: Because there is no client-server split and the software runs locally as a monolith, client prediction is completely inapplicable to this repository.

## Summary
Brimstone is a highly specialized, local-only simulation library built to map game states and generate action diffs (`PowerHistory`) for AI tree searching. It lacks the network transport layer, client-side DTO mapping, and input-translation validation required for a playable multiplayer game, though its `ActionQueue` mechanics for effect resolution are robust.
