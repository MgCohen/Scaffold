# HS_Multilang Architecture Analysis

This document provides an analysis of the `HS_Multilang` repository. Unlike SabberStone, which is a fully-featured, headless C# rules engine, **HS_Multilang is an extremely primitive, client-only Unity prototype** rather than a complete game architecture.

## 1. Core Architecture & Game State

*   **No Central State Machine**: There is no overarching game state machine, rules engine, or task processor. 
*   **Visual-Driven Logic**: The game logic is strongly coupled directly to the Unity UI and GameObjects. Scripts like `cardset.cs`, `hero.cs`, and `dzclient.cs` handle both the data and the visual representation simultaneously.
*   **Entities**: There is no Entity-Component system or modular tag system. Cards are simple visual objects with static data loaded from an XML file (`card.xml`). 

## 2. Client-Server Communication

*   **Network Layer**: It uses Unity's deprecated, legacy networking system (`UnityEngine.Network` and `NetworkView`). 
*   **Connection**: It uses `Network.InitializeServer` to host and `Network.Connect` to join via IP address, bypassing any modern matchmaking or dedicated server logic.

### How Clients Receive Messages
*   Communication is achieved entirely through Remote Procedure Calls (RPCs).
*   Methods are tagged with `[RPC]` (e.g., `[RPC] void OnHero(int c)` or `[RPC] void youfirst()`) and invoked across the network using `NetworkView.RPC(...)`.

### Grouping Messages into Blocks
*   **None**. There is no block logic or event grouping. Commands are atomic and invoke immediately on receipt. There is no concept of complex nested effect resolution.

### Mapping Raw Messages to Objects
*   **None**. Because it uses Unity's high-level `NetworkView.RPC`, primitive types (like `int` for a Hero class enum) are automatically serialized and deserialized by the engine and passed as standard method arguments. There are no structured Data Transfer Objects (DTOs) or packet parsing mechanisms.
*   External account and deck data is fetched via basic HTTP requests (`UnityEngine.WWW`) exchanging string data.

## 3. Player Input & Client Prediction

*   **Input Translation**: Player input directly calls functions or sends RPCs to the hardcoded `NetworkView` targets. There is no translation into an intermediate "Task" or "Command" object before execution. A click on a card simply triggers the card's local Unity `OnMouseUpAsButton()` event.
*   **Client Prediction**: None. The client and server simply hurl RPCs at each other to synchronize visual states (like whose turn it is or what hero was chosen) without any authoritative validation of the rules. 

## Summary
`HS_Multilang` serves as an early visual prototype for a Hearthstone UI in Unity. It lacks a formalized game architecture, rules engine, or modern networking layer, making it unsuitable as a reference for a robust, scalable multiplayer card game backend.
