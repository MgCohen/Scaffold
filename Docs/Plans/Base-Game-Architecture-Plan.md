# Base Game Architecture Plan

Based on the research findings (particularly the deep analysis of `SabberStone`) and the project's specific requirements, here is the proposed architecture plan for the Multiplayer Card Game. This architecture ensures a strict separation between the C# business logic (authoritative server) and the visual presentation layer (client), while natively integrating the project's `CustomSerializableGenerator` and existing `State`/`NetworkMessages` structures.

## Core Architectural Principles

1.  **Strict Client-Server Separation:** The core Game Logic (the rules engine) contains purely functional C# code without any Unity-specific dependencies (`MonoBehaviour`, `GameObject`, etc.) or network-layer coupling.
2.  **Entity-Component-System (ECS) Lite (Tags):** Every object in the game (Cards, Heroes, Players, the Game itself) is an `Entity`. State isn't stored in hardcoded variables but as a dictionary of `GameTags` (e.g., `HEALTH = 30`, `ZONE = PLAY`).
3.  **Command/Task Queue Resolution:** Logic isn't executed instantly via direct method calls. Abstract actions are pushed as discrete `Tasks` into a central `TaskQueue`, which processes them sequentially. This naturally handles cascading events (e.g., Deathrattles, nested triggers).

## Technical Flow Answers

### 1. How does the client receive the data?
The client receives data as a sequential stream of binary network packets over the transport layer. The server's logic engine generates chunks of raw game state modifications (e.g., "Tag X changed," "New Entity created"). These logically grouped events are mapped to Data Transfer Objects (DTOs) defined in `Assets/Scripts/Infra/NetworkMessages` (e.g., `TagChangeMessage`, `ShowEntityMessage`). Before transmission, the `CustomSerializableGenerator` serializes these messages into an optimized binary format to be sent over the wire.

### 2. What the client does with the data?
Upon receiving the binary payload, the client:
1.  **Deserializes** the payload using the generated code from `CustomSerializableGenerator` back into `NetworkMessages`.
2.  **Updates Local State:** The client immediately applies these changes to its local reflection of the `Game` `State` in `Assets/Scripts/Game/State`.
3.  **Queues Visuals:** To prevent the visual presentation from snapping instantly to the new state, the client enqueues these structural differences into a "Visual Action Queue." The UI and Animations consume this queue sequentially (e.g., playing a fireball VFX, then updating the visual health number).

### 3. How the client knows if an action is required?
The server operates on a strict State Machine with specific steps (e.g., `MAIN_ACTION`). When the server enters a step requiring player input, it calculates all legally playable moves and sends an `OptionsBlockMessage` to the client. This customized payload contains a list of valid, abstract integer `Option IDs`. The client unlocking its drag-and-drop or clickable UI elements is entirely dictated by the contents of this `OptionsBlockMessage`.

### 4. How the client sends an action to the server?
Player inputs utilize the **Command Pattern** as "dumb" network requests. When the user interacts (e.g., dragging a spell to an enemy), the client *never* sends a direct instruction like "Deal 3 damage." 
Instead, the client references the pre-approved `OptionsBlockMessage`, finds the corresponding `Option ID` for that play, and generates a simple `SendOptionMessage` containing only the integer indices (e.g., `OptionIndex = 5, TargetId = 12`). It serializes this message via the `CustomSerializableGenerator` and transmits it to the server.

### 5. How the server will receive client actions?
A dedicated infrastructure layer (Network Adapter) sitting outside the core rules engine listens to the TCP/transport socket. It receives the binary payload, deserializes it into the `SendOptionMessage`, and intercepts the packet before it touches the game state. 

### 6. What the server does with client actions?
The server evaluates the abstract `SendOptionMessage`. It correlates the client's requested `OptionIndex` against its authoritative, internal cached list of valid options for that player's current priority. If the packet is validated, the server dynamically translates the index into a concrete, executable C# object (e.g., a `PlayCardTask` or `HeroAttackTask`) and pushes this securely into the `TaskQueue`.

### 7. How the server will move the game state?
The core engine continuously processes the `TaskQueue`. Dequeued `Tasks` execute by exclusively mutating the `GameTags` on relevant `Entities` within `Assets/Scripts/Game/State`. 
For instance, an `AttackTask` doesn't call a complex combat formula directly on a character class; it calculates damage and enqueues a `DamageTask`, which simply decreases the `HEALTH` `GameTag` of the target entity. This purely data-driven mutation defines the movement of the game state.

### 8. How the server will handle cascading effects?
Nested interactions are managed natively by the `TaskQueue` and an observer `TriggerManager`. 
When the rules engine actively modifies a `GameTag` (e.g., a minion's health reaches 0, changing its `ZONE` tag to `GRAVEYARD`), the core `Game` state fires an internal event. The `TriggerManager` listens for these events, iterating over all active entities to find any that possess identical trigger hooks (e.g., "Whenever a minion dies"). If a match is found, the trigger enqueues a *new* `Task` into the existing `TaskQueue`. Because tasks execute sequentially and support hierarchical grouping (Blocks), deep chains of triggers resolve safely without creating infinite recursive stack loops.

### 9. How the server decides what the player can do?
During the `Game.Process()` loop, the State Machine evaluates the board at the beginning of any priority phase. It queries the active player's hand and board entities, verifying their tags (`COST` vs available player mana, boolean tags like `EXHAUSTED` or `FROZEN`, and checking for legally targetable enemies). It compiles a comprehensive list of every legal option, bundles this abstract list into an `OptionsBlockMessage`, and broadcasts it to the client.

### 10. How the server sends data to the client?
As the `TaskQueue` evaluates sequences, it tracks the start and end of logical action chains using Blocks (e.g., `BlockStart`, `TagChange...`, `BlockEnd`). 
As modifications occur in real-time within the core logic, an implementation of the observer pattern records them. Once a logical chunk resolves completely, this accumulated history is grouped into `NetworkMessages`, serialized efficiently by the `CustomSerializableGenerator`, and dispatched to the clients.

### 11. How to decouple or interface the "game-logic" from the networking logic?
The architecture enforces an **Adapter/Facade Pattern** to act as a bridge, adhering to the following strict boundary:

*   **`Scaffold.Game.Logic` (Core Engine):** A pure, headless C# class library. It has ZERO dependencies on `NetworkMessages`, Unity Serialization, or external socket definitions. It operates exclusively via abstract `Entities`, `GameTags`, and `Tasks`.
*   **`Scaffold.Game.Adapter` (Infrastructure Layer):** This wrapper directly interfaces the core engine with the network stack.
    *   **Sending Data:** The Adapter subscribes to standard C# events emitted by the core engine (e.g., `OnHistoryLogGenerated`). When fired, the Adapter converts these functional logs into structural `NetworkMessages` and routes them to the serialization and transport layers.
    *   **Receiving Data:** The Adapter intercepts incoming `NetworkMessages` from the transport layer, decodes them, and manually invokes the core engine's API to construct and inject `Tasks`.

This absolute decoupling ensures the core Game Engine remains fundamentally testable, functional, and purely focused on card game mechanics.
