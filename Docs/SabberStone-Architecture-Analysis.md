# SabberStone Architecture Analysis

This document provides a general analysis of the SabberStone repository, a Hearthstone simulator in C#. It outlines the core architecture and main systems used to replicate a complex card game.

## 1. Core Game State Machine (`Game.cs` & `GameEventManager.cs`)

The heart of SabberStone is a robust state machine driven by the `Game` class.

*   **`Game` Class**: This acts as the central hub and state machine processor. It holds all the current entities, players, and manages the execution flow `Process()`. It delegates specific logic depending on the current phase of the game.
*   **Step Enum**: The game flow is divided into discrete `Step` values (e.g., `BEGIN_FIRST`, `BEGIN_MULLIGAN`, `MAIN_READY`, `MAIN_ACTION`, `MAIN_END`). This defines the strict order of operations for a turn.
*   **`GameEventManager`**: This class observes the `Game` state changes (specifically changes to the `NEXT_STEP` tag) and immediately invokes the corresponding methods on the `Game` class (e.g., calling `game.MainReady()` when the step changes to `MAIN_READY`). This observer pattern separates the flow logic from the state progression logic.

## 2. Entity and Tag System (`Entity.cs` & `Playable.cs`)

SabberStone does not model game objects with hardcoded, discrete properties (like integer health, integer attack directly on a Minion class). Instead, it uses a flexible Entity-Component approach based on "Tags".

*   **`Entity` Base Class**: Every interactive object in the game (Heroes, Minions, Spells, Weapons, and even the Players and the Game itself) inherits from `Entity`.
*   **GameTags**: An `Entity` is essentially a container for a collection of properties called `GameTag`s (which directly correspond to Hearthstone's internal tags, like `ATK`, `HEALTH`, `ZONE`, `COST`, `EXHAUSTED`, `FROZEN`).
*   **`Playable`**: Cards that can be held in a hand and played onto the board inherit from `Playable` (which inherits from `Entity`). `Playable` adds logic for targeting and cost management.
*   **State through Tags**: State changes occur by modifying an entity's tags rather than changing explicit properties. When a tag gets modified (e.g. `HEALTH` goes down), the change fires events that the game systems listen to.

## 3. Rules, Events, and Tasks Engine (`TaskProcessor.cs` & `TriggerManager.cs`)

To handle the incredible complexity of nested card effects, triggered abilities, and chain reactions, SabberStone uses a Command/Task Queue pattern.

*   **Tasks and Actions**: Every action in the game (Drawing a card, Playing a minion, Dealing damage) is encapsulated into a discrete `Task` object. These tasks are enqueued rather than executed instantly.
*   **`TaskQueue` / `TaskProcessor`**: All operations go through the `TaskQueue`. This queue ensures that events resolve in the correct order. It supports "pending tasks", which handles the nested resolution of effects (for example, minion A takes damage, queueing minion A's death, which queues minion B's "whenever a minion dies" trigger, which queue's minion B dealing damage, etc.). The queue naturally manages these cascading triggers safely.
*   **`TriggerManager`**: This is a centralized registry that listens to events from the `TaskProcessor` or `Game` and processes "Triggers" (e.g., `OnDamageTrigger`, `OnSummonTrigger`, `OnDeathTrigger`). If an entity on the board has an effect that triggers when damage is dealt, the `TriggerManager` detects the damage event, finds the active triggers, and enqueues new Tasks into the `TaskQueue` to resolve those triggered effects.

## Summary

SabberStone's architecture is highly decoupled and relies heavily on three design patterns:
1.  **State Machine:** For strict game-phase flow control.
2.  **Tag-based Entities (ECS-lite):** To flexibly describe thousands of unique cards without writing thousands of unique classes. Data is data, and mechanics are driven dynamically by tags.
3.  **Command/Task Queue:** To safely handle deep, chained interactions, death processing (which, similar to MTG's state-based actions or Hearthstone's Aura updates, need strictly ordered resolution logic without recursive stack overflows).

## 4. Client-Server Communication and Messaging (`PowerHistory.cs`)

SabberStone is primarily a simulator, but it closely mirrors Hearthstone's protocol structure for game state updates. 

### How Clients Receive Messages

The server communicates state changes to the client by producing a linear list of `PowerHistory` entries. A unified stream of events, tags, and entity reveals is generated asynchronously as tasks are processed by the game engine. These entries effectively recreate the entire game state sequentially.

### Grouping Messages into Blocks

To handle complex simultaneous logic (like an AoE spell hitting multiple targets and triggering cascading deathrattles), the server uses **Blocks** to logically group operations together.
*   **`PowerHistoryBlockStart`**: Signifies the beginning of an effect resolution. It includes a `BlockType` (e.g., `POWER`, `TRIGGER`, `ATTACK`, `PLAY`).
*   **`PowerHistoryBlockEnd`**: Signifies the end of that specific event tree.
*   **Nesting Logic**: Blocks can be deeply nested. If a `TRIGGER` block causes another event (like Drawing a card), a new `BlockStart` is appended, the card's data is updated, and then a `BlockEnd` closes the inner block. This is natively managed by the `TaskQueue`: whenever a `Task` starts executing, a `BlockStart` is logged; when the `Task` completes, a `BlockEnd` is logged.

### Mapping Raw Messages to Objects

The raw data packets directly map to structural Data Transfer Objects (DTOs) on the client side. They implement the interface `IPowerHistoryEntry` and correspond to the exact packet structures from Hearthstone:
*   **`PowerHistoryTagChange`**: Replicates a single changed `GameTag` for a specific Entity.
*   **`PowerHistoryFullEntity`**: Used when a completely new entity is created from scratch, sending the base entity ID and all its default tags.
*   **`PowerHistoryShowEntity`**: Used when a previously hidden entity (e.g., a card in the opponent's hand) is revealed. It updates the client with the card ID and visible tags.
*   **`PowerHistoryHideEntity`**: Used when a visible card is shuffled into a deck or returned to an opponent's hand.

## 5. Player Input Management (`PlayerTask.cs`)

Player input strictly follows the **Command Pattern**.

Instead of the client directly instructing the game state to modify values (e.g., directly calling `target.Health -= 3`), the client submits atomic requests as **Player Tasks**.
*   **`PlayerTask` Base Class**: Defines any action a user can voluntarily take during their priority.
*   **Concrete Implementations**: The repository maps inputs directly to classes like `PlayCardTask`, `HeroAttackTask`, `MinionAttackTask`, `EndTurnTask`, and `ChooseTask` (for Discover mechanics).
*   **Execution Flow**: When a user drags a spell onto an enemy minion, the client constructs a target structured `PlayCardTask(Controller, SpellCard, TargetCharacter)`. This task is queued into the `TaskProcessor` which checks its legality, and if valid, processes the sequence. This approach perfectly decouples the UI interaction handler from the game rules engine.

## 6. Deep Dive: Network Layer & Kettle Extension

While SabberStone is intrinsically a local C# logic library (`SabberStoneCore`), network support is implemented through an extension called **`SabberStoneKettle`**. 

### Raw Network Implementation
The network layer heavily avoids Unity's high-level networking abstractions in favor of raw C# components:
*   **TCP Sockets**: Connections are made using `System.Net.Sockets.Socket` configured for TCP (`SocketType.Stream, ProtocolType.IP`). 
*   **Stream Wrappers**: Inside `KettleAdapter.cs`, the socket feeds into a `NetworkStream`, which is then read via `BinaryReader` and `BinaryWriter`.
*   **Custom Packets**: Data isn't transmitted as standard `NetworkMessage` types typical in game engines; instead, raw bytes are buffered and parsed back into specialized structures called `KettlePayloads` (which map easily to JSON or byte streams).

### Translating Client Input to Server Tasks
The client *does not* construct `Tasks` directly. `Tasks` are a strictly server-side authoritative concept.
1.  **Client-Side Creation**: When the client takes an action, they formulate a generic DTO indicating their selection. For example, they craft a `KettleSendOption` packet detailing the index of the card played, the sub-option, and the integer ID of the target.
2.  **Transmission**: This generic packet goes through the TCP socket.
3.  **Server-Side Translation**: The server receives this via `KettleSession.OnSendOption(...)`. Before running it through the game rules, the server passes the raw packet into a utility formatter (`Util.KettleOptionToPlayerTask`).
4.  **Task Processing**: This utility correlates the integer IDs in the message to the actual authoritative `IPlayable` and `ICharacter` entities existing in the simulated server state, instantiating the correct `PlayerTask` (e.g. `PlayCardTask`), which is *then* passed securely to the `TaskProcessor`.

### Client Prediction
**SabberStone implements strictly zero client-side prediction.** 

Looking at the `KettleAISession` (the dummy client connected to the network):
1.  The client receives an `OnOptionsBlock` packet from the server (a list of all legally playable cards and targets for that turn).
2.  The client selects an option, builds a `KettleSendOption` packet, sends it, and then goes completely idle.
3.  The client state does not visually or logically mutate upon sending the command. It waits passively until the server receives the command, processes it through the `TaskQueue`, and streams back an `OnHistory` payload (the `PowerHistory` blocks).
4.  Only upon receiving these history blocks does the client actually mutate its visual state.

This "dumb terminal" architecture prevents state desynchronization completely, shifting 100% of the game logic and prediction burdens entirely onto the central server processing stack.
