# Figma to Unity: Import pipeline for UGUI prefabs

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

Repository policy for ExecPlans is defined in `PLANS.md` at the repository root. This document must be maintained in accordance with that file.


## Purpose / Big Picture

Today, building Unity UI from a Figma design file is a manual, error-prone process: a designer exports images and dimensions, a developer recreates the hierarchy by hand, and any design change starts the process over. The goal of this work is a `com.scaffold.figma` package that accepts a JSON file describing a Figma file's node tree and produces a Unity prefab with a correct RectTransform hierarchy, Auto Layout groups, text components, and image fills — no manual re-entry.

The full pipeline has two halves: a *source* side (something that reads Figma and emits JSON) and a *consumer* side (the Unity importer that reads that JSON and builds prefabs). This plan covers the consumer side only. The source side — an MCP client or Figma plugin that calls `get_metadata` — is out of scope for the first pass; it can be added as a separate package later. The Unity importer is valuable on its own because any JSON file that conforms to the schema (hand-authored, generated by a script, exported by a plugin) will produce a correct prefab.

You can see this working when you drop a `*.figma.json` file into the `Assets/` folder, Unity automatically creates a prefab, and that prefab contains a RectTransform hierarchy that mirrors the node tree in the JSON file — including correct anchor presets for each node's Figma constraints, horizontal or vertical layout groups for Auto Layout frames, and TextMeshPro components for text nodes.

## Decision Log

- Decision: single package `com.scaffold.figma`, not two (`com.scaffold.figma.schema` + `com.scaffold.figma.import`). Runtime DTOs live in `Runtime/`; the ScriptedImporter lives in `Editor/`. The assemblies are `Scaffold.Figma` (runtime) and `Scaffold.Figma.Editor` (editor-only, `includePlatforms: ["Editor"]`).
  Rationale: The schema is Editor-only in practice because nothing reads Figma JSON at game runtime. Splitting into two packages adds a dependency to manage and a second `package.json` for negligible benefit. One package with the standard Editor/ layout matches every other Scaffold package.
  Author: Design session, 2026-05-02.

- Decision: The MCP connector (code that calls the Figma MCP server and writes the `.figma.json`) is out of scope for this plan. It will live in a future `com.scaffold.figma.connector` package or a standalone CLI tool if/when needed.
  Rationale: The importer is independently useful: any `.figma.json` file — hand-authored, generated by a REST script, or exported by a plugin — is consumable. Coupling the importer to the MCP client would block v1 on Figma API access and rate limits, which the prior research flagged as a real cost gate. Decoupling also means the connector can be replaced or extended without touching the importer.
  Author: Design session, 2026-05-02.

- Decision: v1 uses `JsonUtility`. v2 switches to Newtonsoft (`com.unity.nuget.newtonsoft-json`) when design tokens introduce a `Dictionary<string, T>` field, which `JsonUtility` cannot deserialize. The switch is a planned migration, not a fallback.
  Rationale: `JsonUtility` is available in all Unity 6 projects without extra dependencies and is sufficient for the v1 schema (no dictionaries, no polymorphic arrays, no top-level arrays). Adding Newtonsoft now is unnecessary cost. By the time v2 lands, the v1 code will exist as a known-working migration target rather than a greenfield design under an unfamiliar serializer.
  Author: Design session, 2026-05-02.

- Decision: `FigmaImporter` produces an imported asset via `AssetImportContext.AddObjectToAsset` + `SetMainObject` on a root `GameObject`. It does **not** write a sidecar `.prefab` via `PrefabUtility.SaveAsPrefabAsset`. The output is not a Prefab Mode-editable asset; it is an imported asset that behaves like a prefab when dragged into a scene.
  Rationale: Auto-rebuild on JSON change is the central value of `ScriptedImporter`. Calling `PrefabUtility.SaveAsPrefabAsset` from inside `OnImportAsset` triggers `AssetDatabase` re-entry, is fragile, and is an established Unity anti-pattern. Designers re-export from Figma rather than hand-editing the imported asset; if a developer needs to customize it (rename a child, add a script), they instantiate it in a scene and save a real prefab from that instance — explicit, one-time, and outside the import path.
  Author: Design session, 2026-05-02.

- Decision: Solid-fill rendering uses a single shared 1×1 white sprite stored at `Runtime/Resources/Scaffold.Figma.WhitePixel.png`, loaded via `Resources.Load<Sprite>("Scaffold.Figma.WhitePixel")` and assigned to every `Image.sprite`. The fill's RGBA goes to `Image.color`.
  Rationale: `UnityEngine.UI.Image` requires a non-null `sprite` to render — `color` alone draws nothing. Generating a sprite at import time and `AddObjectToAsset`'ing it bloats every imported asset with a duplicate texture. A single shared `Resources/` lookup is one line in `StyleMapper` and produces correct rendering for every solid-fill node. `Resources/` is mildly discouraged in favor of Addressables in modern Unity, but the placeholder is referenced only by Editor-only import code, so the discouragement does not apply.
  Author: Design session, 2026-05-02.

- Decision: v1 text rendering uses `TMP_Settings.defaultFontAsset` for every text node and ignores `style.fontFamily`. The feature table marks "font auto-download by Figma font name" as v2.
  Rationale: Every Unity 6 project with TextMeshPro imported has a default font asset, guaranteed to exist. Resolving `style.fontFamily` to a project asset in v1 requires a name-resolution layer, project-wide font scanning, and a "font not found" failure mode — none of which is core to the importer's v1 promise (correct hierarchy and layout). v1 designs will look typographically wrong but structurally correct, which is the expected v1 trade.
  Author: Design session, 2026-05-02.

- Decision: The spike in Milestone 1 is a required gate before Milestone 2. If the Figma MCP output does not contain Figma constraints (LEFT/RIGHT/CENTER/SCALE) in `get_metadata` — only the position/size — the anchor mapping cannot be driven from MCP alone and must fall back to REST API data or be approximated. This changes the schema. Confirm constraints are present before finalising any DTO shape.
  Rationale: The prior research audit explicitly flagged constraints as "should be present but only position/size are promised." A wrong assumption here ripples into every downstream mapper. One real call resolves the ambiguity cheaply.
  Author: Design session, 2026-05-02.

- Decision: The reference implementations to mine are `figma-spec-mcp` (by zaferdace, ~600 lines, `map-to-unity.ts`) for the constraint-to-anchor mapping logic and LayoutGroup wiring, and `UnityFigmaBridge` (by simonoliver) for the Unity Editor/importer-side patterns. Neither is adopted as a dependency; only the mapping logic and patterns are referenced. Both are confirmed broken or incomplete for our use case in the prior research.
  Rationale: `figma-spec-mcp` has the cleanest UGUI-targeted mapper but no Unity importer. `UnityFigmaBridge` has the most complete Unity importer but is REST/batch-only with no JSON intermediate, uses an incompatible architecture, and has drifted toward dormant. We take the best of each as reading material only.
  Author: Design session, 2026-05-02.


## Known Unknowns / Spike Verifications

Two technical assumptions in this plan cannot be confirmed from documentation alone and must be validated against real Figma data during Milestone 1. Each has a defined acceptance test. Failure of either test is a scope-changing event and must be recorded in Surprises & Discoveries with the chosen response.

**Spike Acceptance A — Stretched-anchor coordinate math is correct.**

The constraint→anchor table in the "Figma constraints to Unity anchor presets" section is correct for pinned anchors (where `anchorMin == anchorMax`). For stretched anchors (`LEFT_RIGHT`, `TOP_BOTTOM`, `LEFT_RIGHT` + `TOP_BOTTOM`), the conversion from Figma's absolute rect to Unity's `anchoredPosition` + `sizeDelta` is well-known but footgun-prone: `sizeDelta.x` carries the negative sum of left and right margins, not the element width.

The fixture `sample.figma.json` must include at least one node whose constraints are `{ horizontal: "LEFT_RIGHT", vertical: "TOP_BOTTOM" }` and whose parent has a non-trivial size. The test in `ImporterTests.cs` must:

1. Import the fixture.
2. Programmatically resize the parent `RectTransform` to a different width and height.
3. Force a layout rebuild via `Canvas.ForceUpdateCanvases` and `LayoutRebuilder.ForceRebuildLayoutImmediate`.
4. Assert that the stretched child's runtime `rect.size` and `rect.position` match the proportional Figma intent (the child should grow with the parent, maintaining its margins).

Pass: the assertion holds. The math is correct, the table stands, no further action needed.

Fail: the assertion is off by a constant or scales wrongly. The math is wrong. Stop, debug `ConstraintMapper.Apply` against Unity's `RectTransform.anchoredPosition` semantics for stretched anchors, update the table and the mapper together, and re-run before proceeding to Milestone 4.

**Spike Acceptance B — `get_metadata` returns Figma constraints.**

The prior research flagged that the Figma MCP tool docs explicitly promise `id`, `name`, `type`, position, size, and hierarchy from `get_metadata`, but do not explicitly promise constraints (`LEFT`, `RIGHT`, `CENTER`, `SCALE`, `LEFT_RIGHT`, `TOP_BOTTOM`). Constraints exist in the underlying Figma file format and are returned by the REST API and Plugin API, but their presence in `get_metadata` output is unverified.

During Milestone 1, make one real `get_metadata` call against a Figma frame whose children have non-default constraints. Inspect the raw XML or sparse-text output. Two outcomes:

Pass — constraints are present in `get_metadata` output: schema is unchanged, the future `com.scaffold.figma.connector` package is a pure MCP-to-JSON translator, and the connector plan is straightforward.

Fail — constraints are absent: the schema is unchanged (the importer still consumes constraints), but the future connector cannot be MCP-only. It must combine `get_metadata` (for hierarchy, position, size) with a Figma REST API call (for constraints), correlating by node `id`. Add a note to the v2 entry "Figma MCP client" in the feature coverage table: "requires REST API token in addition to MCP". Record the finding in Surprises & Discoveries with the raw output snippet as evidence.

This verification does not block v1 — the v1 importer consumes a fixture that already has constraints transcribed correctly. Only the v2 connector plan changes shape based on the outcome.

## Context and Orientation

This section explains the relevant package layout, the Figma data model, the Unity UGUI anchor model, and Auto Layout semantics. Read it fully before writing any code.


### Package layout

The package lives at `Assets/Packages/com.scaffold.figma/`. Its internal layout follows the same convention as every other Scaffold package:

    Assets/Packages/com.scaffold.figma/
      package.json
      Runtime/
        Scaffold.Figma.asmdef          (no Unity Editor dependency, autoReferenced: true)
        Schema/
          FigmaDocument.cs
          FigmaNode.cs
          FigmaRect.cs
          FigmaConstraints.cs
          FigmaAutoLayout.cs
          FigmaStyle.cs
      Editor/
        Scaffold.Figma.Editor.asmdef   (includePlatforms: ["Editor"])
        Import/
          FigmaImporter.cs             (ScriptedImporter)
          NodeMapper.cs
          ConstraintMapper.cs
          AutoLayoutMapper.cs
          StyleMapper.cs
      Tests/
        Scaffold.Figma.Tests.asmdef
        Fixtures/
          sample.figma.json            (committed golden file from the spike)
        SchemaTests.cs
        ImporterTests.cs

The `Runtime/` assembly holds only plain C# data-transfer objects (DTOs) and zero Unity engine types. The `Editor/` assembly holds the `ScriptedImporter` and all mapper classes; it compiles only in the Editor and is excluded from player builds. The `Tests/` assembly references both and is also Editor-only.


### The Figma data model

A Figma file is a tree of **nodes**. Every node has:

- `id` — a unique string like `"123:456"`.
- `name` — the layer name as shown in the Figma left panel.
- `type` — one of `DOCUMENT`, `CANVAS`, `FRAME`, `GROUP`, `RECTANGLE`, `ELLIPSE`, `TEXT`, `COMPONENT`, `INSTANCE`, `VECTOR`, or similar.
- `absoluteBoundingBox` — `{ x, y, width, height }` in Figma's coordinate space (origin top-left, y increases downward).
- `constraints` — how the node resizes relative to its parent: `{ horizontal: "LEFT" | "RIGHT" | "CENTER" | "SCALE" | "LEFT_RIGHT", vertical: "TOP" | "BOTTOM" | "CENTER" | "SCALE" | "TOP_BOTTOM" }`.
- `layoutMode` — present on FRAME nodes with Auto Layout enabled: `"HORIZONTAL"` or `"VERTICAL"` (or `"NONE"`).
- `itemSpacing`, `paddingLeft`, `paddingRight`, `paddingTop`, `paddingBottom` — Auto Layout spacing values, present when `layoutMode != "NONE"`.
- `fills` — array of fill objects. A solid fill looks like `{ type: "SOLID", color: { r, g, b, a } }`.
- `style` — text properties when `type == "TEXT"`: `{ fontSize, fontFamily, fontWeight, textAlignHorizontal, textAlignVertical }`.
- `characters` — the text content when `type == "TEXT"`.
- `children` — array of child nodes.

The Figma MCP's `get_metadata` tool returns a sparse XML representation of this tree. For the spike, the output must be manually transcribed into JSON using the schema below. In later milestones, the MCP connector will do this automatically.

When `get_metadata` is called, request a specific frame or component by ID. Do not attempt to walk the entire document in one call — the response will be truncated. For the spike, one frame containing at least one Auto Layout row, one text node, and two nodes with different constraint combinations is sufficient.


### The JSON schema

The committed fixture `Tests/Fixtures/sample.figma.json` is the canonical example. Its top-level shape is:

    {
      "documentName": "My Design",
      "nodes": [ <FigmaNode>, ... ]
    }

A `FigmaNode` is:

    {
      "id": "123:456",
      "name": "Card",
      "type": "FRAME",
      "rect": { "x": 0, "y": 0, "width": 320, "height": 200 },
      "constraints": { "horizontal": "LEFT_RIGHT", "vertical": "TOP" },
      "layoutMode": "VERTICAL",
      "layoutSpacing": 8,
      "layoutPadding": { "left": 12, "right": 12, "top": 16, "bottom": 16 },
      "fills": [ { "type": "SOLID", "r": 1, "g": 1, "b": 1, "a": 1 } ],
      "text": null,
      "style": null,
      "children": [ <FigmaNode>, ... ]
    }

A text node has `"type": "TEXT"`, a non-null `"text"` field with the string content, and a non-null `"style"` field:

    "style": { "fontSize": 14, "fontFamily": "Inter", "color": { "r": 0.1, "g": 0.1, "b": 0.1, "a": 1 }, "alignH": "LEFT", "alignV": "TOP" }

Fields that are not applicable to a node type are null or omitted. The importer must treat missing fields as their defaults (no constraints → anchor preset CENTER, no layoutMode or layoutMode "NONE" → no LayoutGroup, no fills → no Image component).


### Figma constraints to Unity anchor presets

Unity RectTransform anchors control how a UI element repositions and resizes when its parent changes size. `anchorMin` and `anchorMax` are both `Vector2(0..1, 0..1)` in normalised parent space. Common presets and their Figma equivalents:

| Figma horizontal | Figma vertical | anchorMin | anchorMax | pivot | Notes |
|---|---|---|---|---|---|
| LEFT | TOP | (0, 1) | (0, 1) | (0, 1) | Top-left pin |
| RIGHT | TOP | (1, 1) | (1, 1) | (1, 1) | Top-right pin |
| LEFT | BOTTOM | (0, 0) | (0, 0) | (0, 0) | Bottom-left pin |
| RIGHT | BOTTOM | (1, 0) | (1, 0) | (1, 0) | Bottom-right pin |
| CENTER | CENTER | (0.5, 0.5) | (0.5, 0.5) | (0.5, 0.5) | Centered |
| LEFT_RIGHT | TOP | (0, 1) | (1, 1) | (0.5, 1) | Horizontal stretch, top pin |
| LEFT_RIGHT | BOTTOM | (0, 0) | (1, 0) | (0.5, 0) | Horizontal stretch, bottom pin |
| LEFT_RIGHT | CENTER | (0, 0.5) | (1, 0.5) | (0.5, 0.5) | Horizontal stretch, center |
| LEFT_RIGHT | TOP_BOTTOM | (0, 0) | (1, 1) | (0.5, 0.5) | Full stretch |
| LEFT | TOP_BOTTOM | (0, 0) | (0, 1) | (0, 0.5) | Vertical stretch, left pin |
| RIGHT | TOP_BOTTOM | (1, 0) | (1, 1) | (1, 0.5) | Vertical stretch, right pin |
| CENTER | TOP_BOTTOM | (0.5, 0) | (0.5, 1) | (0.5, 0.5) | Vertical stretch, center |
| SCALE | SCALE | (0, 0) | (1, 1) | (0.5, 0.5) | Proportional scale (use stretch + runtime scale) |
| CENTER | TOP | (0.5, 1) | (0.5, 1) | (0.5, 1) | Top-center pin |
| CENTER | BOTTOM | (0.5, 0) | (0.5, 0) | (0.5, 0) | Bottom-center pin |

When `anchorMin == anchorMax` (a pinned anchor), the `RectTransform.anchoredPosition` is the offset from the anchor in screen pixels, and `sizeDelta` is the element's size. When `anchorMin != anchorMax` (a stretched anchor), `anchoredPosition` is the offset added to the calculated center, and `sizeDelta` carries the margin offsets: `sizeDelta.x = -(left_margin + right_margin)`, `sizeDelta.y = -(top_margin + bottom_margin)`.

`ConstraintMapper.cs` must implement a method `Apply(FigmaNode node, RectTransform rt, RectTransform parent)` that sets `anchorMin`, `anchorMax`, `pivot`, `anchoredPosition`, and `sizeDelta` correctly for both the pinned and stretched cases. Compute the node's rect in parent-local space using `node.rect` (absolute) minus `parent.rect` (absolute) top-left, then convert to Unity coordinates (flip y because Unity y-up, Figma y-down).


### Auto Layout to LayoutGroup

When `layoutMode == "HORIZONTAL"`, add `HorizontalLayoutGroup` to the node's GameObject. When `layoutMode == "VERTICAL"`, add `VerticalLayoutGroup`. In both cases:

- Set `spacing` to `layoutSpacing`.
- Set `padding.left`, `padding.right`, `padding.top`, `padding.bottom` from `layoutPadding`.
- Set `childAlignment` to `UpperLeft` as a safe default.
- Set `childForceExpandWidth` and `childForceExpandHeight` to `false` initially (can be refined later when primary-axis sizing modes are added).
- Add `ContentSizeFitter` to the same GameObject with `horizontalFit = Unconstrained` and `verticalFit = Unconstrained` (prevents fight between ContentSizeFitter and fixed RectTransform size when Auto Layout frames have explicit sizes in Figma).

`AutoLayoutMapper.cs` must implement `Apply(FigmaNode node, GameObject go)`.


### ScriptedImporter

`FigmaImporter.cs` must be decorated with:

    [ScriptedImporter(1, "figma.json")]
    public class FigmaImporter : ScriptedImporter

The `version` integer (first argument) must be incremented whenever the importer logic changes in a way that requires existing imported assets to be re-imported. Unity uses this to decide whether to re-run the importer on next domain reload.

The entry point is `OnImportAsset(AssetImportContext ctx)`. The steps are:

1. Read `ctx.assetPath` via `File.ReadAllText`.
2. Deserialise to `FigmaDocument` using `JsonUtility.FromJson<FigmaDocument>`.
3. Create a root `GameObject` named after `document.documentName`.
4. Walk the node tree recursively, calling `NodeMapper.CreateNode(node, parentTransform)` for each node.
5. Save the root as a prefab asset using `ctx.AddObjectToAsset("root", rootGameObject)` and `ctx.SetMainObject(rootGameObject)`.

`NodeMapper.CreateNode` creates a `GameObject`, adds a `RectTransform`, calls `ConstraintMapper.Apply`, calls `AutoLayoutMapper.Apply` if applicable, calls `StyleMapper.Apply` for fills and text, then recurses into children.

The importer does not use `PrefabUtility.SaveAsPrefabAsset`. It uses `AssetImportContext.AddObjectToAsset`, which is the correct API for `ScriptedImporter`. The resulting asset behaves like a prefab in the project.


### Coordinate system conversion

Figma: origin top-left, x right, y down. Unity UGUI: origin varies by anchor, but the canonical Canvas space has origin bottom-left, x right, y up.

When computing a node's position for its RectTransform from Figma absolute rects:

1. Convert the node's Figma rect to parent-local space: `localX = node.rect.x - parent.rect.x`, `localY = node.rect.y - parent.rect.y` (both in Figma's y-down space).
2. Flip y for Unity: `unityLocalY = parent.rect.height - localY - node.rect.height`.
3. The result is the top-left corner in parent-local Unity space. Offset by anchor to get `anchoredPosition`.

This conversion must be done in `ConstraintMapper.Apply` and is not the responsibility of the node walker.


### Testing strategy

Every milestone has tests before it is marked complete. The test assembly references `UnityEditor.TestRunner` and uses `[UnityTest]` for importer tests (which require domain reload) and `[Test]` for pure C# schema and mapper tests.

Schema tests (`SchemaTests.cs`) verify that `sample.figma.json` round-trips through `FigmaDocument` deserialization without data loss: check node count, a specific node's name and type, a specific node's constraints, and a text node's style fields.

Mapper tests verify `ConstraintMapper` and `AutoLayoutMapper` in isolation by constructing `FigmaNode` objects directly and asserting on the resulting `RectTransform` fields.

Importer tests verify the full pipeline by importing `sample.figma.json` via `AssetDatabase.ImportAsset` and loading the resulting prefab, then asserting that the prefab hierarchy matches the expected depth and that a named node has the expected component.

Run all tests with:

    .agents/scripts/validate-changes.cmd

from the repository root. All tests must pass before a milestone is marked complete.
